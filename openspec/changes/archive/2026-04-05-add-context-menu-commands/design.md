## Context

File Cooker 当前主要通过命令面板暴露能力，并按数据来源组织（当前文件链接、剪贴板、搜索结果、Dataview）。用户可以执行批量移动、重命名、属性编辑、删除、合并、同步 flomo、添加到 Canvas 等操作，但在右键操作链路中的可发现性较弱。

本次变更将新增编辑器与文件的右键菜单入口，并保持原有命令面板行为不变。该变更属于跨模块的交互入口扩展：调用入口变化，但核心动作语义与确认流程保持一致。

## Goals / Non-Goals

**Goals:**
- 在编辑器与文件右键菜单中提供高频 File Cooker 操作入口。
- 通过分组子菜单控制信息密度，降低菜单噪音。
- 复用既有 Action 与确认弹窗流程，确保不同入口下行为一致。
- 通过上下文判断控制可见性/可执行性，避免无效操作触发。

**Non-Goals:**
- 不重写现有 Action 实现或确认弹窗逻辑。
- 不移除或重命名现有命令面板命令。
- 本次不引入资源管理器多选文件批处理入口。
- 不新增 File Cooker 现有能力之外的全新操作类型。

## Decisions

1. **在右键菜单下使用统一顶层分组与子菜单结构**
   - **Decision：** 在上下文菜单中新增统一的 `File Cooker` 顶层分组，并按目标类型细分子菜单（选区、当前文件链接、单文件）。
   - **Rationale：** 保持右键菜单可扫描性，避免大量平铺条目影响使用体验。
   - **Alternatives considered：**
     - 全部平铺在根菜单：点击路径短但噪音大，否决。
     - 拆成多个顶层分组：结构分散且不一致，否决。

2. **命令面板与右键菜单并行保留**
   - **Decision：** 保留现有命令注册，同时新增右键菜单注册与分发。
   - **Rationale：** 保持向后兼容，兼顾键盘流与鼠标流用户。
   - **Alternatives considered：**
     - 将能力迁移到右键菜单为主：会破坏已有使用习惯，否决。

3. **为文件右键引入“目标适配”而非复用当前文件读取器副作用**
   - **Decision：** 为文件右键场景增加轻量目标适配路径，将被右键文件转换为可复用的 Action 输入模型。
   - **Rationale：** 现有架构偏 Reader 驱动，文件右键需要“显式目标”输入，避免强依赖“当前激活文件”带来的副作用。
   - **Alternatives considered：**
     - 强制切换当前激活文件后复用 CurrentFileReader：隐式行为过强，稳定性差，否决。
     - 在 file-menu 分支复制一套 Action 逻辑：维护成本高，否决。

4. **高风险操作保留但采用安全排序与确认机制**
   - **Decision：** 删除等高风险操作继续可用，但放在组内后位，并复用现有确认弹窗。
   - **Rationale：** 在效率与安全之间取得平衡，减少误触成本。
   - **Alternatives considered：**
     - 右键菜单完全隐藏高风险操作：限制了高级用户工作流，否决。

## Risks / Trade-offs

- **[Risk] 不同 Obsidian 版本上下文菜单 API 细节差异** → **Mitigation：** 对上下文对象与目标对象增加健壮判空，并在最低支持版本上进行回归验证。
- **[Risk] 上下文可用性判断不一致（无选区、无活动文件等）** → **Mitigation：** 集中管理可执行条件，确保菜单显示与执行前判断一致。
- **[Risk] 命令面板与右键双入口带来认知重复** → **Mitigation：** 使用稳定命名与一致分组，降低理解成本。
- **[Risk] 文件右键目标适配偏离 Reader 输出约定** → **Mitigation：** 定义统一的 ActionModel 构造约束，并通过行为测试对齐现有命令路径结果。
