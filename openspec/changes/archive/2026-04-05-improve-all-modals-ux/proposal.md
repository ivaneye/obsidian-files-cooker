## Why

当前插件中的多个弹窗在视觉与交互上存在明显不一致：信息层级弱、按钮主次不清、危险操作缺少显著语义、长列表场景可读性差。这导致批量操作时认知负担较高并增加误操作风险；随着功能增加，继续按“逐个弹窗手写”会放大维护成本，因此需要尽快做一次统一的弹窗体验优化。

## What Changes

- 对现有全部业务弹窗进行统一体验改造（不新增业务功能），覆盖标题区、说明区、摘要区、列表区、操作区的结构规范。
- 统一操作语义与按钮层级：仅主确认操作使用高强调样式；取消操作降级；删除等高风险操作使用危险语义与明确文案。
- 统一列表呈现策略：显示处理数量、限制列表高度并支持滚动，优化长路径与批量文件场景下的可读性。
- 统一空态与反馈文案：No files 场景、成功/取消提示、校验失败提示采用一致表达风格与术语。
- 补充弹窗样式与复用约定，减少后续新增弹窗的重复实现与风格漂移。
- 为弹窗行为与展示规则补充自动化测试，按 TDD 模式确保回归稳定性。

验收标准：
- 所有现有业务弹窗均采用统一布局骨架与按钮语义。
- 危险操作与普通操作在视觉与文案上可明确区分，且无语义混淆。
- 批量文件场景下弹窗可稳定展示且不出现内容溢出导致的不可操作问题。
- 相关测试通过并覆盖关键交互分支（确认、取消、空态、输入校验）。

## Capabilities

### New Capabilities
- （无）

### Modified Capabilities
- `context-menu-commands`: 调整命令触发后的弹窗交互要求与展示规范，使批量文件操作弹窗具备一致的结构、语义和反馈行为。

## Impact

- 影响代码范围：`src/modal/*` 中现有确认类与输入类弹窗、相关 action/command 的调用链，以及 `styles.css` 的弹窗样式定义。
- 影响测试范围：需补充或更新与弹窗交互相关的单元测试与回归测试。
- 外部 API / 数据结构：不涉及新增外部接口与数据协议。
- 主要风险：
  - 统一样式后与 Obsidian 主题兼容性不足（深浅主题差异）。
  - 批量弹窗改造引入行为回归（按钮事件、校验提示、流程串联）。
- 缓解措施：
  - 采用“原生增强”策略，优先复用 Obsidian 语义样式能力并进行多主题验证。
  - 按弹窗类型分批改造并以测试覆盖关键路径，逐步收敛回归风险。
