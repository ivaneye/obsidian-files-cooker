## MODIFIED Requirements

### Requirement: 菜单分组与安全性
插件 MUST 将 File Cooker 右键能力组织为分组子菜单，并对高风险操作保留确认机制。

#### Scenario: 用户查看右键菜单结构
- **WHEN** 用户打开编辑器或文件右键菜单
- **THEN** File Cooker 条目 MUST 以分组子菜单呈现，而非大量平铺在根菜单

#### Scenario: 用户触发高风险操作
- **WHEN** 用户从右键菜单触发删除等高风险 File Cooker 操作
- **THEN** 插件 MUST 复用现有确认弹窗，确认后才执行实际变更

#### Scenario: 高风险操作弹窗具有危险语义与明确主次
- **WHEN** 用户打开删除等高风险操作的确认弹窗
- **THEN** 弹窗 MUST 以危险语义突出确认操作，并将取消操作呈现为次要操作，避免主次混淆

#### Scenario: 批量操作时弹窗展示统一摘要信息
- **WHEN** 用户打开任一批量文件处理弹窗（如删除、移动、创建、合并、同步）
- **THEN** 弹窗 MUST 展示统一摘要信息（至少包含处理数量与目标上下文），并保证在长列表场景下操作区始终可达

## ADDED Requirements

### Requirement: 业务弹窗统一结构规范
插件内现有业务弹窗 MUST 使用统一的信息结构，包括标题区、说明区、摘要区、内容区和操作区，以确保不同操作之间的认知一致性。

#### Scenario: 用户在不同操作间切换弹窗
- **WHEN** 用户分别打开删除、移动/复制、创建、重命名、属性编辑、合并、同步 flomo 等弹窗
- **THEN** 各弹窗 MUST 采用一致的布局分区与交互位置，用户无需重新学习操作路径

#### Scenario: 文件列表内容过多
- **WHEN** 弹窗内待处理文件数量较多或路径较长
- **THEN** 内容区 MUST 使用受控高度与滚动策略，且不遮挡确认与取消操作按钮

### Requirement: 弹窗操作语义与反馈一致性
插件 MUST 在全部业务弹窗中统一主次操作语义与反馈规范，确保确认、取消、空态和校验失败行为可预期且可观察。

#### Scenario: 用户执行普通确认类操作
- **WHEN** 用户在非危险弹窗中执行确认
- **THEN** 弹窗 MUST 使用主操作语义呈现确认按钮，取消按钮 MUST 保持次要语义

#### Scenario: 用户取消任意业务弹窗
- **WHEN** 用户点击取消关闭弹窗
- **THEN** 插件 MUST 提供一致风格的取消反馈，并且 MUST 不执行任何文件变更

#### Scenario: 弹窗输入校验失败
- **WHEN** 用户提交缺失必填输入或不合法输入
- **THEN** 插件 MUST 阻断后续执行并显示明确、可操作的校验提示

#### Scenario: 数据为空时打开弹窗
- **WHEN** 弹窗输入数据为空（如无可处理文件）
- **THEN** 弹窗 MUST 展示统一空态说明与安全关闭操作，且 MUST 不触发业务执行
