## Why

当前 File Cooker 已支持基于搜索结果、当前文件链接和 Dataview 查询结果进行批量文件操作，但尚未覆盖用户在 Bases 插件中的查询结果场景。随着 Bases 使用频率提升，用户需要在同一工作流内直接对 Bases 查询命中的“对应文件”执行批量操作，并保持与现有命令体验一致。

## What Changes

- 新增针对 Bases 查询结果的批量处理能力，目标对象明确为“可映射到 Vault 文件的结果”。
- 新增 Bases 结果读取链路，并复用现有 Action 体系（移动、同步 flomo、合并、删除、复制链接、属性编辑、重命名、添加到 Canvas、Task 添加到 Canvas）。
- 提供与现有 Dataview/搜索命令一致的交互入口与命名风格，降低学习成本。
- 对无法映射到文件的 Bases 行执行“忽略并提示数量”，避免中断整批操作。
- 明确不包含对 Bases row 本身的字段编辑能力（非本次范围）。

## Capabilities

### New Capabilities
- `bases-query-file-operations`: 支持将 Bases 查询结果作为输入源，并对可映射文件执行与现有批量流程一致的文件操作。

### Modified Capabilities
- （无）

## Impact

- Affected code:
  - `main.ts`（命令注册入口扩展）
  - `src/command/`（新增或扩展 Bases 命令组）
  - `src/reader/`（新增 Bases 结果读取器）
  - `src/modal/`（复用现有确认与选择弹窗，必要时补充提示信息）
  - `tests/`（新增 Bases 场景下的命令与读取行为测试）
- Dependencies:
  - 依赖 Bases 插件可用上下文或可解析结果来源。
- 用户影响面：
  - 用户可在 Bases 结果场景直接执行批量文件操作，减少手工中转步骤。

### 验收标准

- 在 Bases 查询结果场景下可触发完整文件操作命令集（与现有 Dataview 范围一致）。
- 仅对可映射文件执行操作，不对 row 数据本体做修改。
- 非文件结果被忽略并给出明确数量提示。
- 现有 Dataview、搜索结果、当前文件相关能力无回归。

### 主要风险与缓解

- 风险：Bases 查询结果结构在不同上下文不一致，导致解析失败。
  - 缓解：统一抽象 Reader 适配层，增加失败与边界测试，提供清晰提示。
- 风险：命令入口增多导致用户认知混乱。
  - 缓解：保持命名规范一致，统一前缀与描述。
- 风险：批量操作误伤。
  - 缓解：复用现有确认弹窗与摘要信息展示机制。

### 实施与验证约束

- 涉及行为变更的实现将采用 TDD（先失败测试、再最小实现、再重构）并补充自动化测试覆盖。
