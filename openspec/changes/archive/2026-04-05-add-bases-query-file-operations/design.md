## Context

当前插件以“Command + Reader + Action”分层组织批量处理能力：
- Command 负责注册命令入口与上下文可用性判断；
- Reader 负责从不同来源（Search/CurrentFile/Dataview/Clipboard）收集目标；
- Action 负责对目标文件执行批量操作。

本次变更新增 Bases 查询结果作为“目标来源”，且用户已明确：操作对象必须是“对应文件”，不包含对 Bases row 字段本身的编辑。

约束与相关方：
- 约束：需保持现有命令体系与交互风格一致，避免用户心智割裂。
- 约束：需兼容 Bases 结果中“无法映射到文件”的数据，处理策略为“忽略并提示数量”。
- 相关方：File Cooker 用户（命令面板高频用户）、维护者（需控制回归风险）。

## Goals / Non-Goals

**Goals:**
- 增加 Bases 查询结果的文件批处理入口，覆盖与 Dataview 同等级的完整操作集。
- 复用既有 Action 与弹窗确认机制，保持一致体验与低回归成本。
- 建立统一的映射与过滤策略：仅处理可映射文件，忽略非文件项并提示数量。
- 在实现阶段采用 TDD，并通过自动化测试覆盖主流程、边界与失败场景。

**Non-Goals:**
- 不提供 Bases row 字段级写入、更新、删除能力。
- 不改变现有 Dataview/Search/CurrentFile 的行为语义。
- 不引入新的外部持久化存储或远程服务。

## Decisions

### 决策 1：新增 BasesReader，沿用 Reader 接口接入现有 Action 管线
- 方案：新增 `BasesReader`（实现 `Readable`），输出 `ActionModel[]`，对接已有 Action。
- 选择理由：
  - 与现有 `DataviewReader` / `SearchResultsReader` 架构一致，改动边界清晰；
  - 可最大化复用现有危险操作确认、目标选择弹窗、批量限制逻辑。
- 备选方案：在现有 Reader 中混入 Bases 分支。
  - 未选原因：职责混杂，后续维护与测试隔离更差。

### 决策 2：第一阶段仅支持“可映射到 vault 文件”的 Bases 结果
- 方案：对 Bases 返回结果做路径提取与文件存在校验，仅将可解析文件送入 Action。
- 选择理由：
  - 与用户目标完全一致（操作文件而非 row）；
  - 明确边界，降低错误写入 row 数据的风险。
- 备选方案：同时支持 row 级动作。
  - 未选原因：能力边界变化过大，不符合本次范围。

### 决策 3：非文件结果采用“忽略并提示数量”而非中断
- 方案：在执行前汇总不可映射项数量，提示后继续处理可映射文件。
- 选择理由：
  - 更符合批处理场景容错预期；
  - 与用户指定策略一致（“忽略并提示数量”）。
- 备选方案：遇到不可映射项即失败退出。
  - 未选原因：对混合数据集不友好，用户需要重复清洗输入。

### 决策 4：命令集“全量对齐 Dataview”，命名风格一致
- 方案：Bases 命令覆盖移动/同步 flomo/合并/删除/复制链接/属性编辑/重命名/加文件到 Canvas/加任务到 Canvas。
- 选择理由：
  - 与用户“全要、全部、需要统一风格”要求一致；
  - 降低迁移成本，用户可按同一语义在不同来源间切换。
- 备选方案：只做最小子集（如移动+删除）。
  - 未选原因：无法满足用户明确范围。

### 决策 5：测试策略采用 TDD，先 Reader 再 Command 集成
- 方案：
  - 单元层：`BasesReader` 映射、过滤、提示逻辑；
  - 集成层：命令注册可用性、触发后调用 Reader+Action 链路；
  - 回归层：确保既有 Dataview/Search 命令不受影响。
- 选择理由：行为变更集中在入口与解析层，TDD 能最快暴露语义偏差。

### 模块交互示意（Mermaid）

```mermaid
flowchart LR
    U[用户在 Bases 结果上下文触发命令] --> C[BasesCommand]
    C --> R[BasesReader]
    R --> M{结果是否可映射为文件}
    M -- 是 --> F[构造 ActionModel(file)]
    M -- 否 --> I[计入 ignoredCount]
    F --> A[既有 Action: move/delete/merge/...]
    I --> N[Notice: 已忽略 N 条非文件结果]
    A --> X[复用既有确认弹窗与执行流程]
```

## Risks / Trade-offs

- [风险] Bases 结果结构在不同视图/版本下不一致 → [缓解] 设计适配层与健壮判空逻辑，并增加多形态 fixture 测试。
- [风险] 全量命令引入导致注册复杂度上升 → [缓解] 复用 DataviewCommand 的命令组织方式与命名约定。
- [风险] 用户误以为会修改 row 字段 → [缓解] 在命令说明与提示中强调“仅处理对应文件”。
- [权衡] 容错继续执行提高可用性，但用户可能忽略被过滤项 → [缓解] 提示包含明确数量，并在确认摘要中体现。

## Migration Plan

- 发布步骤：
  1. 增加 Bases 命令与 Reader，并补充测试；
  2. 更新 README（中英文）说明“仅文件操作”边界与忽略策略；
  3. 发布后观察用户反馈与错误日志（若有）。
- 回滚策略：
  - 若出现兼容性问题，可通过移除/禁用 Bases 命令注册快速回退，不影响既有来源能力。

## Open Questions

- Bases 插件在当前项目环境下可稳定读取结果的官方入口是否固定（API/视图 DOM/上下文对象）？
- “任务添加到 Canvas”在 Bases 场景下的任务文本来源是否与 Dataview TASK 语义完全等价？若不等价，是否允许降级提示为“不适用”。

## 测试策略

- 测试框架：沿用项目现有 `vitest`。
- 测试分层：
  - Reader 单元测试：路径提取、去重、不存在文件过滤、ignoredCount 提示；
  - Command 集成测试：命令注册完整性、触发条件与行为调用；
  - 回归测试：Dataview/Search 相关测试全量通过。
- Mock/Stub 策略：
  - Mock Obsidian `app/workspace/vault/Notice`；
  - Stub Bases 返回结果数据集（全文件、混合、全非文件、空结果）。
- 验证目标：每个 Spec Scenario 对应至少一个自动化测试用例。

## 代码组织与注释策略

- 新增实现与测试代码需补充最小必要注释：
  - 实现代码注释说明职责边界（“只处理文件映射，不处理 row 写入”）；
  - 测试代码注释说明测试目的、输入场景与预期结果。
